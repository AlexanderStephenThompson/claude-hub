# Personalized AI Context System
## A Practical System for Non-Generic Output

---

## Core Principle

AI doesn’t lack intelligence — it lacks *context continuity*.

Your goal is not better prompts.  
Your goal is to build a **persistent operating system for your AI**.

Think:
- Prompt = one-off instruction  
- Context system = onboarding + memory + standards  

---

# 1. The Minimum Viable System (Start Here)

If you do nothing else, build these 4 files:

### 1. `identity.md`
Who you are + how you think

### 2. `voice.md`
How you communicate

### 3. `standards.md`
How you build (code + structure)

### 4. `examples.md`
What “good” and “bad” look like

That alone solves ~70% of your problem.

---

# 2. High-Impact Context Categories (Ranked)

## Tier 1: Massive Impact (Do First)

### Identity (identity.md)
Defines how the AI should think

Include:
- Role: Information designer
- Core principle: clarity > cleverness
- Preferences: systems, structure, semantic thinking
- Hard avoids: (e.g. help desk work, generic fluff)
- Decision bias: simple → scalable → systemized

---

### Voice System (voice.md)
This is what kills “AI tone”

Structure:
- Tone traits: direct, structured, practical
- Sentence style: short-medium, varied
- Openings: concrete, not abstract
- Banned words: leverage, synergy, etc.
- Formatting preferences

Example:
We do:
- Clear structure
- Practical framing
- Real examples

We don’t:
- Over-explain basics
- Use vague business language
- Sound corporate

---

### Golden Examples (examples.md)
Most people skip this — biggest mistake

Include:
- 5–10 “this is perfect” outputs
- 2–3 “this is wrong” outputs

Format:
## Good Example
[content]

Why it works:
- Clear structure
- Specific
- Sounds like me

## Bad Example
[content]

Why it fails:
- Generic tone
- No structure

This trains *judgment*, not just rules.

---

### Standards (standards.md)

Split into:

#### Code Standards
- Tech stack (React, Tailwind, etc.)
- Naming conventions
- File structure
- Patterns you use/avoid

#### Thinking Standards
- Prefer systems over hacks
- Prefer reusable over one-off
- Prefer clarity over abstraction

---
